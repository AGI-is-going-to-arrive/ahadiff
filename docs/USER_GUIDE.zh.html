AhaDiff User Guide · 2026-05-25 · English

知返 AhaDiff 用户使用指南

AhaDiff 把一次 git diff 变成带代码证据的课程、声明清单、测验、复习卡、概念图谱和质量棘轮。它默认是 local-first：每个仓库自己的 .ahadiff/ 是真相源。

新增视频教程：打开中文介绍和使用教程 MP4；打开 English tutorial MP4。两个视频分别使用对应语言界面和烧录字幕；中间字幕源文件不随公开文档树发布。

1. 快速开始

安装命令

# 当前还未发布到 PyPI；在源码 checkout 中安装 CLI
uv tool install --editable .

# 或者直接在源码目录运行
uv run ahadiff --version

源码 checkout

uv sync --locked --dev
uv run python -m ahadiff --version
uv run python -m ahadiff doctor

第一次在仓库中使用

cd /path/to/your/repo
ahadiff init
ahadiff doctor
ahadiff config show --resolved

AhaDiff 是 per-repo 工作方式。生成物写在当前仓库的 .ahadiff/，不是全局数据库。

SQLite 说明：当前存储门禁接受 SQLite 3.51.3+，以及补丁回移分支 3.50.4+ 和 3.44.6+。这覆盖 uv tool 安装的 Python 3.11.14 自带 SQLite 3.50.4 的情况。

2. 配置 GPT-5.5 Provider

不要把真实 key 写进 Git、README、manifest 或命令行历史。推荐把 key 放到环境变量里，只把变量名交给 AhaDiff。

macOS / Linux

export AHADIFF_PROVIDER_BASE_URL="https://api.example.com/v1"
export AHADIFF_PROVIDER_API_KEY="<provider-api-key>"

ahadiff provider test   --name gpt55   --provider-class openai_responses   --base-url "$AHADIFF_PROVIDER_BASE_URL"   --model gpt-5.5   --api-key-env AHADIFF_PROVIDER_API_KEY   --privacy-mode explicit_remote

Windows PowerShell

$env:AHADIFF_PROVIDER_BASE_URL="https://api.example.com/v1"
$env:AHADIFF_PROVIDER_API_KEY="<provider-api-key>"

ahadiff provider test --name gpt55 --provider-class openai_responses --base-url $env:AHADIFF_PROVIDER_BASE_URL --model gpt-5.5 --api-key-env AHADIFF_PROVIDER_API_KEY --privacy-mode explicit_remote

这里使用 openai_responses，是因为 GPT-5.5 有两种 registry profile：普通 openai 接入按 40 万上下文预算，openai_responses / API 接入按 105 万上下文预算。endpoint 如果通过 live probe 上报可信 total context，可以覆盖这两个内置 profile。

配置完成后，单个 provider 或 Settings 里选好的生成 provider/model 会被 ahadiff learn 直接使用。日常学习命令不需要再写 provider 参数；显式 --provider / --model 只用于临时覆盖：

ahadiff learn --last
ahadiff learn --last --provider gpt55 --model gpt-5.5

如果本地 OpenAI-compatible 服务上报的 JSON 能力不准确，可以在 .ahadiff/config.toml 里只覆盖已知布尔能力：

[providers.local.capability_overrides]
supports_native_json_schema = false

NewAPI 默认关闭 supports_native_json_schema；如果你的 NewAPI 网关后端真的支持 native JSON schema，可在 provider config 加 capability_overrides = { supports_native_json_schema = true }。

在 Settings 里，provider 卡片可以用未保存的草稿 provider class、model name 和可选 model_limits_name 预览模型上限。这个预览不会调用远端 provider，也不会读取 API key。max_output_tokens 留空表示 Auto；可信的已知硬上限会在保存时自动收紧并返回 warning，未知、低置信度、route-specific 或 local-runtime 上限只显示 warning，不会写成确定保证。

3. 生成一次学习 run

选择 diff 来源
工作区、未暂存、已暂存、最近一次提交、提交范围、时间窗口、patch 文本或 stdin、patch URL 和文件对比都支持全平台。递归目录对比目前需要 macOS/Linux 的安全目录文件描述符。
生成课程和证据
AhaDiff 会捕获 diff、扫描安全风险、生成 claims / lesson / quiz / score 等 artifact。
打开 WebUI 学习
用 Lesson 看解释，用 Diff 查证据，用 Quiz 检验理解，用 Review 做间隔复习。
ahadiff learn --last
ahadiff learn --last --open
ahadiff learn HEAD~1..HEAD
ahadiff learn --since "2 hours ago"
ahadiff learn --staged
ahadiff learn --unstaged --include-untracked
ahadiff learn --unstaged --include-untracked --changed-path src/app.py
ahadiff learn --patch change.diff
ahadiff learn --patch-url "https://example.com/change.diff"
ahadiff learn --compare old.py new.py
ahadiff learn --compare-dir old/ new/
ahadiff learn --last --against-spec SPEC.md
ahadiff learn --last --against-spec SPEC.md --spec-semantic-review

Dashboard 的“开始学习”对话框和这些命令保持同一套心智模型。四个快速卡片对应工作区、未暂存、已暂存和最近一次提交；更多捕获来源按 Git 高级、补丁来源、文件对比分组，每张卡片都会显示对应 CLI 形态。路径范围只对工作区类运行生效，且只接受仓库相对路径。WebUI 的补丁输入框只接受粘贴的 unified diff 文本，限制为 4096 字节；补丁文件、stdin 或更大的补丁请用 CLI。远程 patch URL 会走和 CLI 一样的安全检查；loopback 和私有网络 URL 会被拒绝。目录对比在缺少安全目录文件描述符遍历的平台上会关闭失败。耗时较长的 CLI 命令会显示 Rich 状态提示；它只是终端进度反馈，不是给脚本解析的稳定输出契约。

7. 功能状态：默认、Opt-in、依赖项

功能状态说明
Lesson / Claims / Quiz / Score默认主链运行 ahadiff learn 后生成，具体产物取决于 diff 和 learnability。Quiz 的源码证据会在作答后再展示。
可选 LLM judge可选；确定性评分仍是主口径成功时写入 judge.json；失败时写入有界、脱敏的 judge_failure.json，只包含 provider、model、错误类型和安全错误消息。缺少 judge artifact 时 API 返回 404，不会变成 500。
测验题量默认固定；可手动改成自适应默认生成 3 题。固定模式可设置为 1 到 30 题。可在 Settings 里开启，或临时使用 --quiz-mode auto；自适应模式会根据 diff stats 选题量，默认范围是 3-12 题。
结构化 JSON 输出默认 JSON object；带 1 次 validation retry公开 artifact 形状不变。只有 provider 能力声明支持时才使用 native schema；不支持的 mode 会降级。截断或格式不完整的 fallback JSON 会触发重试，不会被直接接受。
自适应捕获上限新配置自动；自定义过则手动自动模式会根据 provider probe、内置模型表、输出预留、安全预留和 CJK diff 密度计算捕获上限。旧配置只要改过 capture.max_files、capture.hard_limit 或 capture.max_patch_bytes，就会迁移为手动模式。运行时 patch 读取仍封顶 50 MiB。
Provider 智能配置Settings 中可用Provider 卡片会按当前草稿 provider class、model 和可选 limits profile 预览保存后的模型上限，不会在每次编辑时远程探测。它也会展示 provider 原生 thinking 支持、低置信度 warning、推荐的 max-output 值，以及保存 API 返回的收紧提示。Settings 会分别展示 generate 和 judge 模型上限摘要。
WebUI / Welcome / Dashboard / Diff / Review默认可用通过 ahadiff serve 本地打开。Welcome 的 Before/After demo 会把过长的原始 diff 折叠到右侧课程内容高度，显示已展示 / 总行数；短 diff 或空 diff 不显示折叠控件。
AI 工具指引Settings 写入；Guide 只读Settings 会把 15 个安装目标分成 CLI / IDE / CI，展示本地化快速开始步骤、示例提示词、预期输出、平台说明和确定性的 FastAPI middleware 演示。Guide 会在默认折叠的 Agent Skills 卡片中展示同一套使用提示和 manifest 预览，但不提供写入/移除按钮；打印模式会展开这些卡片，方便阅读。
GPT-5.5 LLM需要配置配置 provider、base URL、model 和 API key 环境变量后使用。
Challenge默认关闭需要配置 opt-in。CLI 当前提供 build/status，完整推进在 WebUI/API。
APKG 导出依赖 extra需要 ahadiff[anki]；当前可导出 active review cards，但 stable namespace GUID 仍未落地。
Watch依赖 extra需要 watchdog 相关依赖；否则可以手动运行 learn。
Graphify refresh依赖外部产物/缓存graph refresh 和 WebUI refresh 会把已有 graphify-out/graph.json 重新导入 AhaDiff 的图缓存和 review database；它们本身不会运行 graphify update。learn 阶段在 Graphify CLI 存在时可先尝试 update，再导入新产物。WebUI 刷新会按后端长刷新窗口等待，最长 600 秒。导入受 graph.max_nodes_import 限制（默认 50000，允许范围 1000-50000）；超过上限的刷新会返回 GRAPH_NODE_LIMIT。
Team 功能未发布仍在 roadmap 中，不能写成当前已完成能力。

8. 常见问题

为什么没有 Lesson / Diff / Quiz？

这些页面依赖具体 run_id。先运行一次 ahadiff learn，再从 Dashboard 或命令输出进入对应 run。

Provider 失败怎么办？

ahadiff doctor
ahadiff config show --resolved
ahadiff provider test --name gpt55 --provider-class openai_responses --base-url "$AHADIFF_PROVIDER_BASE_URL" --model gpt-5.5 --api-key-env AHADIFF_PROVIDER_API_KEY --privacy-mode explicit_remote

课程被跳过怎么办？

可能是 diff 太小、learnability 太低或 verified claims 不足。确认这是你想学习的改动后，可以使用 --force-learn。

怎么改测验题量？

到 Settings → Preferences 里设置。固定模式会始终使用配置的题数，范围是 1 到 30；自适应模式会读取本次捕获到的 diff stats，默认范围是 3-12。旧 run 没有这些 stats 时，会回退到固定题数。兼容说明：旧版 AhaDiff 会拒绝保存的大于 10 的题量。

捕获上限怎么决定？

新 repo 默认使用自动捕获。AhaDiff 会优先使用 live provider probe，拿不到时用内置模型表，再拿不到才用保守默认值。total context、input limit 和 output limit 会分开解析；output reserve 只从 input budget 里扣一次。只有 Gemini 这类显式 split-envelope registry rows 会把 input 加 output 存成 total context。手动模式会继续使用你在 config 或 Settings 里保存的数字。在 Settings 里，provider 表单会按未保存的草稿实时预览这些上限，不会远程探测，所以切换 provider class、model 或 limits profile 后，可以先看到提示再保存。

LLM judge 会决定最终通过吗？

不会。最终 verdict 由确定性 score.json 和 hard gates 决定。judge.json 是 advisory，只帮助你理解模型评审意见；如果某个维度不适用，例如没有 spec 的 spec_alignment，Score 和 Judge 都会按 N/A / 0/0 展示。

Accuracy / Evidence 的门禁为什么会变？

Rubric 里的基准阈值仍是 Accuracy 14/20、Evidence 12/18。大 diff 会按可见文件、hunk 和 changed lines 使用自适应阈值，并把 ratio、regime 和 basis 写进 hard_gates.*.policy。这不是放过坏证据；rejected claim 比例过高、contradicted claim、安全问题和真正缺失 evidence 仍会让 run 失败。

Diff Coverage 怎么判断？

Diff Coverage 只使用持久化 line_map.json 里的可见文件和 hunk。被省略的文件不会进入分母。hunk 覆盖按 changed lines 加权，同时保留 hunk 数量下限，避免很小的 hunk 被完全稀释。hard gate 使用自适应 claim-anchor 阈值，并会把 ratio、regime 和 visible basis 写进 gate detail。

哪些安全发现会让 run 失败？

未缓解的 Critical safety finding 会让 run 失败。Critical secret finding 只有在本地捕获记录里带完整 redaction 结构时才算已缓解，包括 rule、hash、source、line 和 column。

能不能说没有任何 bug？

不能做绝对保证。上一轮完整发布 gate 覆盖了后端 unit/integration/eval、静态检查、wheel、前端 typecheck/build/Vitest、完整 Playwright、real-serve、CLI 行为矩阵、Linux 目标容器测试和 GPT-5.5 live fixture。后续 follow-up 只按各自改动面复跑。最新 compare-dir 安全和 generated skill 清理 follow-up 复跑了后端 unit / integration / eval、ruff / format、pyright、前端 typecheck / Vitest / build、i18n parity 和 git diff check。没有重跑 wheel、real-serve、live judge、远端 CI、Linux Docker gate、真实 Windows runner 或完整 Playwright matrix。
