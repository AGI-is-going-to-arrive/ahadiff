Why in this diff: client.ts:82-104 changed constant delay into backoff+jitter. Remember: Backoff protects the server. Jitter prevents synchronized retries. ``` 这个 compact card 用于右栏 marginalia、SRS 卡片、Graph 节点详情和 Claude Skill  progressive disclosure。([arXiv][4]) --- ## 4.2 Karpathy/autoresearch：纳入“三文件契约 + 单指标 + git ratchet” Karpathy/autoresearch 真实结构很简单：`prepare.py` 固定不改，`train.py` 是 agent  唯一编辑的文件，`program.md` 是 agent 指令；README 还明确说 `program.md` essentially 是一个 super lightweight skill，训练固定 5 分钟，用 `val_bpb` 作为可比较指标。([GitHub][6]) AhaDiff 要完整继承这个哲学，不要写一个过度复杂的“自动优化引擎”。最终形态里保留三文件契约： ```text program.md             人类写的自然语言状态机 evaluator.py           immutable，不允许 agent 改 generator_prompt.md    agent 可以改，用来提升 lesson 质量 ``` AhaDiff 的映射： | autoresearch                                | AhaDiff                                                   | | ------------------------------------------- |  --------------------------------------------------------- | | `prepare.py` fixed evaluator/data utilities | `evaluator.py` +  `evals/rubric.yaml` 固定评估尺子               | | `train.py` agent edits                      | `generator_prompt.md` /  lesson template / repair strategy | | `program.md` agent instruction              | `program.md` 控制 improve  loop                              | | `val_bpb` lower is better                   | `lesson_score` higher is better                           | | 5-minute fixed training budget              | fixed token/cost/time  budget                              | | `results.tsv`                               | `results.tsv`                                             | | keep improved commit, reset worse commit    | keep better lesson,  discard worse lesson                  | `program.md` 可以这样写： ```markdown # AhaDiff Improve Loop You are improving how AhaDiff teaches a git diff. This retry loop assumes idempotency, but the diff does not prove all  callers are idempotent. - Jitter is added, but retry budget is not. - Test coverage only checks success-after-retry, not max failure  exhaustion. ``` 这样才能避免被资深开发者认为“这是初学者玩具”。 --- ## 5.3 Spec-before-code：从“事后补课”升级成“计划-实现-学习闭环” 命令： ```bash ahadiff plan "add OAuth login" ``` 输出： ```text .ahadiff/specs/oauth-login/  SPEC.md  acceptance_tests.md  concept_primer.md  risk_register.md ``` 然后开发完成后： ```bash ahadiff learn HEAD~1..HEAD --against .ahadiff/specs/oauth-login/SPEC.md ``` 最终 lesson 里多一个 section： ```markdown ## Spec Alignment Planned: - callback error handling missing Verdict: CAUTION — implementation covers core flow but not error branch. ``` 这个设计非常重要，因为它解决了 Reddit 对 AntiVibe 类工具的核心质疑：不要只是 AI 写完后事后解释，而要把“先  spec，再实现，再学习”连起来。 --- ## 5.4 Graphify 兼容：不要重复造 repo graph 最终命令： ```bash ahadiff graph import graphify-out/GRAPH_REPORT.md ahadiff learn HEAD~1..HEAD --use-graphify ahadiff graph export --format graphify-overlay ``` AhaDiff 的图只做四类节点： ```text File        文件 Symbol      函数/类/接口 Concept     学习概念 Claim       可验证断言 ``` Graphify 提供 repo-level structure，AhaDiff 提供 diff-level learning  evidence。前端 Graph Explorer 要有一个 toggle： ```text [Repo Map] [This Diff] [Learning Memory] [Unverified Claims] ``` --- ## 5.5 不确定性标记：把“不能证明”变成高级感 每篇 lesson 必须有： ```markdown ## Not Proven by This Diff - The exact performance improvement is not proven by this change. - The author intent cannot be inferred from code alone. - This looks like a security improvement, but no test in the diff proves it. ``` 每条 claim 的状态： ```text verified       能回到 diff hunk weak           有代码线索，但因果不完整 not_proven     diff 无法证明 contradicted   与代码冲突 ``` 这会让用户信任你。AI 文档最大的风险不是写得不好，而是写得太自信。 --- # 6. Claim Verifier 是真正护城河 这是最终系统最核心的技术点。 普通工具输出： > This improves reliability. AhaDiff 必须输出： > This improves retry resilience **only for idempotent calls**,  because `sendWithRetry()` now catches transient network errors and  retries with jitter. Evidence: `src/client.ts:L82-L104`. `claims.jsonl`： ```json {  "claim_id": "c007",  "lesson_id": "abc123.v4",  "text": "The retry loop now uses exponential backoff with jitter.",  "source_hunks": [    {      "file": "src/client.ts",      "start": 82,      "end": 104,      "hunk_hash": "h_9f2a"    }  ],  "changed_symbols": ["sendWithRetry"],  "concepts": ["exponential_backoff", "jitter"],  "status": "verified",  "confidence": 0.91,  "verifier_notes": "Backoff factor and random jitter both appear in  changed hunk." } ``` deterministic verifier 先做： ```text source_hunk 是否存在 file:line 是否在 diff 中 函数名是否真实存在 是否提到 diff 中不存在的 symbol 是否有无证据泛化词：faster / scalable / secure / production-ready 每个 changed symbol 是否至少被解释一次 每个重要 claim 是否有 source ``` LLM judge 只负责更高层判断： ```text 因果是否合理 解释是否有教学价值 例子是否帮助理解 quiz 是否能测试真实理解 ``` 这样可以降低 LLM-as-judge 幻觉风险。 --- # 7. 最终命令体系 不要只做一个 `learn`。最终 CLI 应该围绕“计划、学习、验证、复习、发布”设计。 ```bash # 事前 ahadiff plan "add OAuth login" # 事后学习 ahadiff learn HEAD~1..HEAD ahadiff learn
