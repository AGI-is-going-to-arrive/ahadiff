## 1. concepts.jsonl → SQLite Migration

### Current Code State
- **File**: `src/ahadiff/wiki/concepts.py` (~180 lines)
- **Key functions**: `append_concepts()` (line 58), `load_concepts_page()` (line 148), `_AncestryCache` (line 36)
- **Format**: JSONL append-only file at `<repo>/.ahadiff/concepts.jsonl` + `concepts_local.jsonl`
- **Entry fields**: `concept`, `term_key`, `term`, `display_name`, `lang`, `aliases`, `source_refs`, `branch_hint`, `introduced_by_run`, `updated_by_runs`, `related_claims`, `file_refs`, `parents`, `children`, `relationship_type`
- **Current pagination**: `ConceptPage` dataclass with `entries` + `next_cursor` (line-number based cursor)
- **Branch awareness**: `_AncestryCache` calls `git merge-base --is-ancestor` subprocess per unique `source_ref`, capped at `_MAX_ANCESTRY_CHECKS=200`
- **Serve API**: `GET /api/concepts` in `routes_runs.py:140` wraps `load_concepts_page()` via `anyio.to_thread.run_sync`

### Proposed Approach

**New table in `review.sqlite`** (not a separate DB — keeps single-file truth source):

```sql
CREATE TABLE IF NOT EXISTS concepts (
    concept_id TEXT PRIMARY KEY,        -- UUID7
    term_key TEXT NOT NULL,             -- normalized key for dedup
    concept TEXT NOT NULL,              -- original concept name
    display_name TEXT NOT NULL,
    lang TEXT NOT NULL DEFAULT 'en',
    aliases_json TEXT NOT NULL DEFAULT '[]',
    source_refs_json TEXT NOT NULL,     -- JSON array of commit refs
    branch_hint TEXT,
    introduced_by_run TEXT NOT NULL,
    updated_by_runs_json TEXT NOT NULL, -- JSON array
    related_claims_json TEXT NOT NULL,
    file_refs_json TEXT NOT NULL,
    parents_json TEXT NOT NULL DEFAULT '[]',
    children_json TEXT NOT NULL DEFAULT '[]',
    relationship_type TEXT,
    created_at_utc TEXT NOT NULL,
    updated_at_utc TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS ix_concepts_term_key ON concepts (term_key);
CREATE INDEX IF NOT EXISTS ix_concepts_introduced_run ON concepts (introduced_by_run);
CREATE INDEX IF NOT EXISTS ix_concepts_branch ON concepts (branch_hint);
```

**Migration path**:
1. Add `_migrate_v2_to_v3()` in database.py migration chain (current is v2)
2. On first v3 open: scan existing `concepts.jsonl`, insert rows, set `PRAGMA user_version=3`
3. Keep `concepts.jsonl` write for backward compat (one release cycle), then deprecate
4. `append_concepts()` writes to both SQLite + JSONL during transition
5. `load_concepts_page()` switches to SQL `ORDER BY created_at_utc DESC LIMIT ? OFFSET ?` with cursor-based pagination (using `concept_id` as cursor, consistent with existing `before` pattern in `routes_runs.py`)

**Branch awareness change**: Replace subprocess `git merge-base` with a `branch_visibility` column or precomputed visibility table. For 10k+ entries the O(n) subprocess approach is untenable. Cache ancestry results in a `branch_ancestry` table refreshed on `ahadiff learn`.

### Schema Changes
- `review.sqlite` PRAGMA user_version 2 → 3
- New `concepts` table + indices

### API Surface Changes
- `GET /api/concepts` — no contract change, same cursor pagination, faster response
- Internal: `append_concepts()` and `load_concepts_page()` signatures unchanged

### Estimated Complexity
- ~400-500 lines (migration + new DAO + dual-write + tests)
- ~3-4 days

### Risks
- **Data migration**: Corrupt JSONL lines need graceful skip-and-warn
- **Branch ancestry**: Precomputing requires knowing which branches are active; stale branches may have orphan concepts
- **Backward compat**: Users on v0.2 opening v1.0 DB need friendly error (existing `newer-version` pattern covers this)
- **Concurrent access**: `append_concepts` currently uses atomic temp file write; SQLite gives better concurrency via WAL

### Dependencies
- Must come before FTS5 (item #2) — FTS5 indexes the concepts table

## 2. FTS5 Full-Text Search

### Current Code State
- No FTS5 anywhere in the codebase
- **Searchable content** lives in:
  - `result_events` table: `source_ref`, `weakest_dim`, `note_json` (in `review.sqlite`)
  - `cards` table: `concept`, `display_path`, `symbol` (in `review.sqlite`)
  - `concepts.jsonl` → future `concepts` table: `concept`, `display_name`, `aliases`
  - Run artifacts on disk: `lesson.full.md`, `claims.jsonl`, `quiz.jsonl` (in `runs/<run_id>/`)
  - `learning_signals` table: `signal_type`, `payload_json`

### Proposed Approach

**Three FTS5 virtual tables**:

```sql
-- 1. Concepts FTS (in review.sqlite, after concepts migration)
CREATE VIRTUAL TABLE IF NOT EXISTS concepts_fts USING fts5(
    concept, display_name, aliases_text,
    content='concepts', content_rowid='rowid'
);
-- Triggers to keep in sync with concepts table

-- 2. Cards FTS
CREATE VIRTUAL TABLE IF NOT EXISTS cards_fts USING fts5(
    concept, display_path, symbol,
    content='cards', content_rowid='rowid'
);

-- 3. Lesson/Claims content FTS (new table + FTS)
CREATE TABLE IF NOT EXISTS run_content (
    id INTEGER PRIMARY KEY,
    run_id TEXT NOT NULL,
    content_type TEXT NOT NULL,  -- 'lesson_full', 'lesson_hint', 'claim', 'quiz_question'
    content_text TEXT NOT NULL,
    source_ref TEXT,
    created_at_utc TEXT NOT NULL
);
CREATE VIRTUAL TABLE IF NOT EXISTS run_content_fts USING fts5(
    content_text, content_type,
    content='run_content', content_rowid='id'
);
```

**Indexing strategy**:
- Concepts + Cards: contentless FTS5 with triggers (auto-sync on INSERT/UPDATE/DELETE)
- Lesson/Claims: Indexed at `sync_result_event` time — when a run is finalized, parse lesson markdown and claims JSONL, insert into `run_content`
- **Relevance ranking**: Use FTS5 `bm25()` function, with `content_type` boost weights (concepts > lesson > claims > quiz)

**Query API**:
```
GET /api/search?q=<query>&type=concepts|lessons|cards|all&limit=20&cursor=<id>
```
Returns ranked results with snippet highlighting via `snippet()` FTS5 function.

### Schema Changes
- `review.sqlite` PRAGMA user_version 3 → 4 (after concepts migration)
- 3 new FTS5 virtual tables + 1 `run_content` backing table
- Content-sync triggers

### API Surface Changes
- New: `GET /api/search` endpoint
- Modify: Existing `GET /api/concepts` could accept optional `?q=` for filtered search

### Estimated Complexity
- ~600-800 lines (FTS tables + sync triggers + query API + tests)
- ~4-5 days

### Risks
- **SQLite FTS5 availability**: FTS5 is a compile-time option; need runtime check (`SELECT * FROM pragma_compile_options WHERE compile_options = 'ENABLE_FTS5'`). Standard Python 3.11+ SQLite includes FTS5 on all major platforms, but must verify
- **Index size**: 1000 lessons × 10KB average = 10MB FTS index; manageable for local
- **Stale content**: If lesson is regenerated, `run_content` rows need DELETE + re-INSERT
- **Memory**: FTS5 merge operations can briefly spike memory on large indexes

### Dependencies
- Requires concepts SQLite migration (#1) first
- Must happen before `GET /api/search` API (#5)

## 3. Async DAO for Serve

### Current Code State
- **Serve stack**: Starlette + Uvicorn (in `serve/app.py`)
- **Current pattern**: All route handlers are `async def` but call sync SQLite via `anyio.to_thread.run_sync`
- **Examples**: `routes_runs.py` wraps every DB call: `await anyio.to_thread.run_sync(lambda: load_result_events_page(...))`
- **DB connection**: `connect_review_db()` creates new `sqlite3.Connection` per call, no pooling
- **Write lock**: `serve/lock.py` provides `serve_repo_write_lock()` using portalocker + threading.Lock
- **Routes using threadpool**: `list_runs`, `get_run`, `get_lesson`, `get_claims`, `get_quiz`, `get_diff`, `get_concepts`, `get_ratchet_history`, `get_review_queue`, `post_review_rate`, all signal endpoints

### Proposed Approach

**Option A: aiosqlite (recommended)**
- Replace `sqlite3` with `aiosqlite` in serve routes only
- Keep sync `sqlite3` for CLI paths (no async overhead for CLI)
- Create `serve/async_db.py` with async connection factory + context manager
- Connection pooling: Simple pool of 1 writer + N readers (SQLite WAL supports concurrent reads)

```python
# serve/async_db.py
import aiosqlite

class AsyncReviewDB:
    def __init__(self, db_path: Path, *, max_readers: int = 4):
        self._db_path = db_path
        self._reader_pool: asyncio.Queue[aiosqlite.Connection]
        self._writer: aiosqlite.Connection | None = None

    async def read(self) -> AsyncContextManager[aiosqlite.Connection]: ...
    async def write(self) -> AsyncContextManager[aiosqlite.Connection]: ...
```

**Option B: Keep anyio.to_thread (status quo, enhanced)**
- Add a connection pool for the thread-wrapped sync calls
- Less migration effort but doesn't unlock true async benefits

**Recommendation**: Option A for serve paths. The existing sync DAO functions in `review/database.py` remain untouched — async wrappers live in `serve/async_db.py` and call the same SQL. This avoids a massive refactor of all 1700+ lines in `database.py`.

### Migration path:
1. Add `aiosqlite` dependency
2. Create `serve/async_db.py` with pooled async connections
3. Migrate route handlers one at a time (read routes first, then write routes)
4. Keep `anyio.to_thread.run_sync` as fallback for complex DAO calls during transition
5. Add `async_review_db` to `ServeState` (currently has `review_db_path: Path`)

### Schema Changes
- None (same SQLite schema)

### API Surface Changes
- None externally. Internal: route handlers drop `anyio.to_thread` calls

### Estimated Complexity
- ~500-600 lines (async_db module + route migration + tests)
- ~3-4 days

### Risks
- **aiosqlite maturity**: Well-maintained, used by FastAPI ecosystem; low risk
- **Connection lifecycle**: Must ensure connections are closed on shutdown (Starlette lifespan event)
- **Write contention**: Single async writer with WAL is fine for local-first; no need for queue
- **Testing**: Need async test fixtures (`pytest-anyio` already available)

### Dependencies
- Independent of other items; can be done in parallel with #1/#2
- New dependency: `aiosqlite >= 0.20`

## 4. FSRS Optimizer Self-Adaptive

### Current Code State
- **File**: `src/ahadiff/review/scheduler.py` (~160 lines)
- **FSRS library**: `fsrs` package (py-fsrs), version tracked via `scheduler_version()` → `"fsrs-{version}"`
- **Current weights**: `default_scheduler_parameters()` returns library defaults via `Scheduler().to_dict()["parameters"]`
- **Storage**: `scheduler_presets` table in `review.sqlite`:
  ```
  preset_id TEXT PRIMARY KEY      -- currently only 'default'
  weights TEXT NOT NULL            -- JSON array of FSRS parameters
  desired_retention REAL           -- default 0.9
  scheduler_version TEXT
  total_reviews INTEGER            -- DEFAULT 0 (currently not incremented!)
  last_optimized_utc TEXT          -- currently always NULL
  created_at_utc TEXT
  ```
- **Card review**: `_record_card_review()` fetches weights via `_scheduler_weights_for_card()` from preset, calls `review_fsrs_card()`
- **Review logs**: `review_logs` table stores every review (card_id, rating, reviewed_at_utc, elapsed_days, scheduled_days, state)
- **Missing**: `total_reviews` is never incremented; `last_optimized_utc` is never set; no optimizer code exists

### Proposed Approach (Blueprint 3-stage spec)

**Stage 1: Cold Start (<500 reviews)**
- Use library default weights (current behavior)
- Increment `total_reviews` counter on each `_record_card_review()` call
- No optimizer runs

**Stage 2: Accumulation (500-999 reviews)**
- Run FSRS optimizer on accumulated `review_logs` data
- Use `fsrs.Optimizer` (py-fsrs provides this) to compute personalized weights
- Store result in `scheduler_presets.weights` for preset `'default'`
- Retrain trigger: `total_reviews >= 500` AND (`last_optimized_utc` is NULL OR `days_since >= 30` OR `new_reviews >= max(512, total * 0.5)`)

**Stage 3: Mature (≥1000 reviews)**
- Same optimizer but with higher confidence
- Dual threshold for retrain: `≥30 days since last optimize` OR `new reviews ≥ max(512, 50% of total)`

**Implementation**:

```python
# review/optimizer.py (new file)
from fsrs import Optimizer

class FSRSOptimizerRunner:
    def __init__(self, db_path: Path):
        self.db_path = db_path

    def should_retrain(self) -> bool:
        """Check dual threshold: ≥30 days OR new_reviews ≥ max(512, 50%)"""
        ...

    def run_optimization(self) -> tuple[float, ...]:
        """Load review_logs, run Optimizer, return new weights"""
        ...

    def apply_weights(self, weights: tuple[float, ...]) -> None:
        """UPDATE scheduler_presets SET weights=?, last_optimized_utc=?, total_reviews=?"""
        ...
```

**Schema changes needed**:
```sql
-- Add to scheduler_presets (v3 migration)
ALTER TABLE scheduler_presets ADD COLUMN reviews_since_optimize INTEGER NOT NULL DEFAULT 0;
ALTER TABLE scheduler_presets ADD COLUMN optimizer_stage TEXT NOT NULL DEFAULT 'cold_start';
  -- CHECK (optimizer_stage IN ('cold_start', 'accumulation', 'mature'))
```

**Integration point**: Call `optimizer.should_retrain()` at end of `record_card_review_once()`. If true, schedule optimization (sync for CLI, background for serve).

### API Surface Changes
- New: `GET /api/review/optimizer-status` — returns stage, total_reviews, last_optimized, next_retrain_estimate
- Modify: `GET /api/review/queue` response could include optimizer stage info

### Estimated Complexity
- ~350-450 lines (optimizer module + migration + integration + tests)
- ~3-4 days

### Risks
- **py-fsrs Optimizer API**: Need to verify `fsrs.Optimizer` exists and accepts review log format; if not, implement gradient descent manually (~200 extra lines)
- **Performance**: Optimization on 1000+ reviews takes seconds, acceptable for CLI; for serve, run in background thread
- **Weight divergence**: Personalized weights might cause scheduling anomalies for edge users; add sanity bounds on parameters
- **Backward compat**: Old presets with default weights continue working; optimization is additive

### Dependencies
- `total_reviews` is now verified to increment correctly during unique review recording; FSRS Optimizer is not blocked by that earlier suspected bug.
- Should come after concepts migration (#1) to share the v3 migration

## 6. Section-Level Helpfulness

### Current Code State
- **Lesson schemas** (`lesson/schemas.py`):
  - `LessonFull`: Pydantic model with `title`, `summary`, `sections` (list of `LessonSection`), `claim_ids`
  - `LessonSection`: has `section_id`, `section_type`, `title`, `content`, `claim_ids`, `evidence_refs`
  - `section_type` enum values: `"tldr"`, `"what_changed"`, `"why_it_matters"`, `"claims"`, `"walkthrough"`, `"concepts"`, `"misconceptions"`, `"not_proven"`, `"quiz_preview"`, `"sources"`
- **Helpfulness signal**: `POST /api/signals/helpfulness` exists in `routes_signals.py`
  - Accepts `target_kind`, `target_id`, `payload` (freeform dict)
  - Stores in `learning_signals` table with `signal_type='helpfulness'`
  - Currently no aggregation or consumption of this data
- **V6 UI reference**: Shows section helpfulness scores (e.g., "TL;DR +0.06, Walkthrough +0.19") in the Scaffolding block

### Proposed Approach

**Data model**: Helpfulness signals are already stored; need aggregation + consumption.

1. **Signal payload standardization**:
```json
{
  "target_kind": "lesson_section",
  "target_id": "<run_id>:<section_id>",
  "payload": {
    "helpful": true,          // boolean thumbs up/down
    "section_type": "walkthrough",
    "scaffolding_level": "full"
  }
}
```

2. **Aggregation DAO** (in `review/database.py`):
```python
def get_section_helpfulness_scores(
    db_path: Path,
    *,
    min_signals: int = 5,
) -> dict[str, float]:
    """Aggregate helpfulness by section_type. Returns {section_type: score}."""
    # SQL: GROUP BY section_type, AVG(CASE WHEN helpful THEN 1.0 ELSE 0.0 END)
```

3. **Consumption**: Feed into lesson generation as context — sections with low helpfulness get de-prioritized in `lesson_compact.md` generation.

4. **Serve API**: New `GET /api/stats/helpfulness` endpoint returns per-section-type scores.

### Schema Changes
- No schema change needed; `learning_signals` table already stores this
- Consider adding index: `CREATE INDEX ix_learning_signals_target ON learning_signals (json_extract(payload_json, '$.target_kind'))`
  - Note: SQLite JSON functions require 3.38+; current minimum is 3.51.3, so this is safe

### Estimated Complexity
- ~150-200 lines (aggregation DAO + consumption in lesson generator + API endpoint + tests)
- ~2 days

### Risks
- **Cold start**: No signals initially; need sensible defaults
- **Bias**: Users may only click thumbs-down, skewing scores negative; consider implicit positive (viewed without complaint = neutral-positive)
- **Section ID stability**: If lesson is regenerated, section IDs change; helpfulness scores are per-run, not cross-run

### Dependencies
- Independent; can be done anytime

**watchdog library** (Python file system events):

```python
            self._pending_files.add(Path(event.src_path))
            self._reset_debounce()

    def _should_watch(self, path: str) -> bool:
        """Filter: only tracked files, not .ahadiff/, not .git/"""
        ...

    def _trigger_incremental(self):
        """After debounce, compute diff of changed files and run learn"""
        ...
```

**Which files trigger re-learn**:
- Any tracked file that changes (git-tracked, not in `.ahadiffignore`)
- NOT: `.ahadiff/` directory, `.git/` directory, build artifacts
- Debounce: 5 seconds after last change (configurable)

**Incremental strategy**:
1. Compute `git diff` of changed files only
2. If diff is trivial (< 5 lines, only whitespace), skip
3. Run learn pipeline on the incremental diff
4. Merge new concepts into existing concepts (dedup by `term_key`)
5. Update cards affected by changed files (mark stale if file changed significantly)

**Resource limits**:
- Max 1 concurrent watch-triggered learn
- Max 10 incremental learns per hour (prevent runaway on rapid edits)
- Skip if previous learn is still running
- Configurable: `config.toml` `[watchdog]` section with `enabled`, `debounce_seconds`, `max_hourly`

- New: `GET /api/watch/status` — watcher state

### Estimated Complexity
- ~400-500 lines (watcher + incremental diff + rate limiting + tests)
- ~4 days
- New dependency: `watchdog >= 4.0`

### Risks
- **Platform differences**: watchdog uses inotify (Linux), FSEvents (macOS), ReadDirectoryChangesW (Windows) — all supported but behavior differs
- **Git state**: Watcher sees file changes before they're staged/committed; diff is against HEAD
- **Large repos**: Watching thousands of files; watchdog handles this well with OS-level APIs

#### Q3: `_normalize_id()` fuzzy matching — implementation path

Current `compute_term_key()` is deterministic (NFKC + casefold + slug). It handles:
- Case normalization: "React Hooks" → "react-hooks"
- Unicode normalization: CJK characters → hex code points
- Whitespace/punctuation stripping

**Missing for Graphify matching**:
- Abbreviation expansion: "SQL" vs "Structured Query Language"
- Plural/singular: "hooks" vs "hook"
- Hyphenation variants: "event-loop" vs "event loop" vs "eventloop"

**Proposed `_normalize_id()` for fuzzy matching**:
```python
# wiki/matcher.py (new file)
def fuzzy_match_concept(
    graphify_label: str,
    existing_concepts: dict[str, dict],  # term_key → entry
    *,
    threshold: float = 0.85,
) -> str | None:
    """Match a Graphify label to an existing concept. Returns term_key or None."""
    # Step 1: Exact term_key match
    candidate_key = compute_term_key(graphify_label)
    if candidate_key in existing_concepts:
        return candidate_key

    # Step 2: Normalized token overlap (Jaccard similarity)
    label_tokens = _tokenize(graphify_label)
    best_score, best_key = 0.0, None
    for key, entry in existing_concepts.items():
        concept_tokens = _tokenize(entry["concept"])
        score = _jaccard(label_tokens, concept_tokens)
        if score > best_score:
            best_score, best_key = score, key
    return best_key if best_score >= threshold else None
```

**Complexity**: ~80-120 lines. Jaccard on token sets is sufficient for v1.0; full edit-distance or embedding-based matching is overkill for local-first.

#### Q4: 7-state freshness query — dedicated API endpoint?

**Current state**: Only `"source_present"` is implemented. The design doc references 7 internal freshness states with 4-value external projection, but the code shows a single binary check.

**Designed 7 states** (from integration plan + CLAUDE.md):
1. `fresh` — graph.json newer than HEAD commit
2. `stale` — graph.json older than HEAD commit
3. `missing_source` — no graphify-out/graph.json
4. `missing_imported` — source exists but not imported
5. `source_present` — file exists (current implementation)
6. `corrupted` — file exists but invalid JSON
7. `version_mismatch` — schema version incompatible

**4-value external projection** (for frontend):
- `fresh` → "full" mode
- `stale` / `missing_partial` → "learning_only" mode
- `missing` / `corrupted` / `version_mismatch` → "empty" mode
- `unknown` → "empty" mode (fallback)

**Recommendation**: Yes, add a dedicated endpoint:

#### Q5: _AncestryCache subprocess cap (200) — adequate for large repos?

**Current behavior** (`concepts.py:50`): After 200 unique `git merge-base --is-ancestor` calls, all remaining concepts are assumed visible (`return True`). This is a performance safety valve.

**Analysis for large repos**:
- 200 unique source_refs means 200 subprocess calls (each ~5-20ms) = 1-4 seconds
- A repo with 1000+ runs might have 500+ unique source_refs
- After cap: concepts from unverifiable commits become visible (false positives, not false negatives)
- **Risk**: Low. False-positive visibility is safe (shows extra concepts, doesn't hide real ones)

**Recommendation for v1.0**: With concepts in SQLite (#1), replace subprocess ancestry checks with:
1. **Precomputed visibility table**: At `ahadiff learn` time, bulk-compute `git log --ancestry-path` and store reachable commits in a `commit_ancestry` table
2. **Incremental refresh**: Only check new commits since last refresh
3. **This eliminates the 200 cap entirely** — SQL join replaces subprocess calls

```sql
CREATE TABLE IF NOT EXISTS commit_ancestry (
    commit_ref TEXT PRIMARY KEY,
    is_reachable_from_head INTEGER NOT NULL,  -- 0 or 1
    checked_at_head TEXT NOT NULL,            -- HEAD ref when this was computed
    checked_at_utc TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS ix_commit_ancestry_head ON commit_ancestry (checked_at_head);
```

**Complexity**: ~120-160 lines. Bulk `git log --format=%H` is much faster than per-commit `merge-base`.
