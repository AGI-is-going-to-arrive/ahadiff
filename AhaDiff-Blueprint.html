Diff 完整流程详解

从 vibe coding 到打开学习页面，每一步如何触发、数据如何流转、用户需要做什么。

前置条件 · 什么时候能用？

Prerequisites
Level 3 · Git 仓库（完整功能）
  • 证据绑定 + Symbol 索引
  • 跨文件上下文
  • 棘轮回滚（删除 worktree）
  • Graph overlay + Backlinks
  • 版本追踪（source_ref）
--last / --staged / --since
Level 2 · 文件对比（无需 git）
  • 证据绑定 + 部分 Symbol
  • 跨文件上下文
  • 棘轮回滚
  • Backlinks（限当前 run）
  • Content hash 追踪
--compare old.py new.py
Level 1 · Patch 文件（最轻量）
  • 证据绑定（基于 hunk）
  • Symbol 索引
  • 棘轮回滚
  • Backlinks / Graph
  • Patch hash 追踪
--patch file.patch / stdin
不一定是 git 仓库 — git 是默认主流程（Level 3 完整功能），但通过 --patch 和 --compare 也能在非 git 环境下使用，只是部分功能降级。所有 level 都支持：证据绑定、Lesson 生成、Quiz、SRS 复习。

Capture Modes · 3 Levels
Level 3 · Git（完整功能）
推荐

Level 1-2 · 非 Git（无需 git 仓库）
v0.1 新增
E · Patch 文件 / stdin 管道
ahadiff learn --patch fix.patch --open
不需要 git。从 patch 文件、PR 下载、CI 产出、或 diff a b | ahadiff learn --patch - 管道输入。Level 1 功能。
v0.1.1

不需要 git。IDE 修改了单文件？直接对比新旧版本学习。Level 2 功能。
用户不需要指定行号或文件名 — AhaDiff 自动解析 diff 中的所有变更。所有 Level 都支持 Lesson + Quiz + SRS 核心学习功能。

普通 AI 解释 vs AhaDiff

Why Different
普通 AI 工具
diff → 解释文本
"这段代码加了重试功能，使用指数退避，对 POST 请求也是安全的。"
  • 没有代码证据
  • 可能包含错误说法
  • 不知道哪些是 AI 猜的
  • 看完就忘了
AhaDiff
diff → claims → 验证 → lesson → quiz → review
verified "使用指数退避 + 抖动" — src/client.ts:L76-83
contradicted "对 POST 也安全" — 无 idempotency key，转为误解题
  • 每句话有 file:line 证据
  • 错误说法被标记并变成测验
  • 不能证明的标 not_proven
  • Quiz + SRS 帮你真正记住
§ 01 · 灵感借鉴详解

6 个灵感项目如何接入

每个项目都经过源码验证，明确标注了哪些是直接借鉴、哪些是概念改编、哪些是 AhaDiff 自研。绝不虚构归因。
AK
Karpathy / autoresearch
github.com/karpathy/autoresearch
概念改编
概念改编 直接借鉴
原版核心：program.md + prepare.py（immutable）+ train.py（mutable），agent 修改 Python 训练代码，训练 5 分钟，检查 val_bpb 是否改进，keep 或 discard。README 明确 prepare.py 中的 evaluate_bpb 是 ground truth metric。没有 Phase 2.5 或 stuck 检测，仅在 program.md 中说 "if you feel stuck, think harder"。
AhaDiff 如何借鉴：将 N-文件契约的概念映射到学习场景 —— 评估对象从 Python 代码变为 Markdown prompt，评估指标从单一 val_bpb 变为 8 维 rubric + verdict。results.tsv（导出视图；唯一真相源为 review.sqlite）和 keep/discard 循环是直接借鉴。

program.md→improve_program.md
val_bpb (单指标)→aha_score (8 维 rubric)
results.tsv (5 列)→results.tsv (11 列，导出视图；唯一真相源为 review.sqlite)
keep / discard→keep / discard + git ratchet
DS
Darwin-skill
github.com/alchaincyf/darwin-skill
直接借鉴
直接借鉴
原版核心：SKILL.md 里明确 8 维 rubric（结构 60 + 效果 40 = 100 分），先找得分最低的维度，每轮只提出一个改进方案，编辑 SKILL.md 后重新评估。效果维度需要 spawn 独立子 agent，不能自己评自己。分数提升则 keep，否则 git revert HEAD 回滚。核心逻辑全在 SKILL.md 自然语言指令中（仅有一个辅助截图脚本 screenshot.mjs）。
Phase 2.5 触发条件：连续 2 个 skill 在 round 1 就 break 时触发 structural rewrite（不是 3 个）。这是 darwin-skill 独有的机制，autoresearch 没有此机制。
AhaDiff 如何借鉴：
  • Weakest-dimension-first 策略 → 每轮只改一个维度
  • 生成模型与 judge 模型分离 → Sonnet 生成，Haiku 评估
  • Phase 2.5 structural rewrite → 连续 2 个优化目标首轮即无增益时触发
  • git revert 回滚机制 → 不提升就 discard（AhaDiff 改为 worktree 隔离）
  • 子 agent 独立评测 → 跨模型评估

SkillCompass
github.com/Evol-ai/SkillCompass
借鉴 + 自研
三档判定 阈值调整 8 维自研
原版核心：local-first skill quality evaluator，原版 6 维（Structure / Trigger / Security / Functional / Comparative / Uniqueness）评估 skill 文件质量（非学习笔记质量）。PASS / CAUTION / FAIL 三档判定，D3 Security 作为硬门。阈值为 PASS ≥ 70，FAIL < 50。

S0
SKILL0 (ZJU-REAL)
arxiv.org/abs/2604.02268
概念改编
撤架学习法 粒度扩展
原版核心：Helpfulness-driven Skill Retention。训练阶段从 full skill context 开始，通过 Dynamic Curriculum 根据 helpfulness 逐步撤掉 skill context。Budget 递减方式是阶段跳变 [6, 3, 0]（非线性递减）：Step 1-60 保留全部 6 个 skill，Step 61-120 保留 top-3，Step 121-180 完全撤架。Helpfulness 计算原版是 skill file 级。
AhaDiff 如何借鉴：AhaDiff 不做模型训练（无 RL），借鉴的是"撤脚手架学习法"思想，映射为三段式 lesson：

Step 1-60: Full 6 skills→Day 0: lesson.full.md（完整解释）
Step 61-120: Top-3 skills→Day 1: lesson.hint.md + quiz
Step 121-180: No skills→Day 3+: quiz-only + compact card
File-level helpfulness→Section-level helpfulness（自行扩展）

Nuwa-skill
github.com/alchaincyf/nuwa-skill

v3 新增：ahadiff serve 双适配器

Layer 7a Static + 7b Serve
Layer 7 拆分为 7a Static Snapshot（file:// 模式）和 7b Live Serve（ahadiff serve，Starlette + Uvicorn 轻量本地服务器）。同一套 Jinja2 模板通过 data-mode="serve|static" 属性切换行为，实现 Progressive Enhancement。
CLI → orchestrator → Layer 5/6 ↓ ┌─────────────────────┴─────────────────────┐ ↓ ↓ 7a: Static HTML 7b: localhost:8765 data_bundle.json 内嵌 REST API 实时读 SQLite file:// 可直接打开 Quiz/SRS 答题写回 SQLite
7a · STATIC SNAPSHOT
一次性将 lesson + claims + quiz 序列化为 data_bundle.json，嵌入 HTML。无需服务器，file:// 可直接打开。适合 CI 产物归档。
7b · LIVE SERVE (Task 14.5)
ahadiff serve 启动 Starlette + Uvicorn 本地服务器（默认 :8765，bind=127.0.0.1），通过 Layer 5/6 接口读写 review.sqlite，提供 REST API。默认 read-only，写路由需 token 鉴权。依赖 Task 0（serve_app contract）+ Task 14 + Task 15（review.sqlite schema）。
路由鉴权分级：读路由无需 token（GET /api/runs、/api/locale 等）；写路由必须验 X-AhaDiff-Token header（POST /api/signals/*、PUT /api/locale）。Token 在 serve 启动时自动生成并注入 data_bundle。
安全三重保障：bind 127.0.0.1（仅回环，拒绝外网）+ Host/Origin/Referer 校验（只允许 localhost）+ 写路由 token 鉴权。

三层锁矩阵

CLI 与 serve 安全共存
获取顺序严格为 repo_write → db_write → serve_write，不可逆序，防止死锁。CLI 写入和 serve 写入都通过分层锁保护并发安全。GET 请求不拿锁，只依赖 SQLite busy_timeout。
锁实现粒度持有者说明
repo_write_lockPID lockfile仓库级CLI（learn/improve）保证同一仓库同一时刻只有一个写入会话；含 liveness check（进程存活检测）
db_write_lockSQLite WAL + busy_timeoutSQLite 级ratchet layer / serve依赖 SQLite 自身的 WAL 模式 + busy_timeout=5000ms 实现跨进程并发写保护；CLI 和 serve 共享同一 SQLite 文件
serve_write_lockasyncio.Lock进程内serve endpoint保护 serve 进程内写操作的并发安全（异步上下文）
获取顺序图： repo_write_lock → db_write_lock → serve_write_lock (PID file) (WAL + flock) (asyncio.Lock) ↑ 仅 serve 进程内需要

Q1: Diff 是怎么检测的？和 IDE 的 LSP/AST 有什么不同？

Codex 验证
AhaDiff 的方式 · Git Unified Diff
@@ -68,7 +68,29 @@ class ApiClient - async request(path) { - return fetch(path) + async request(path, opts) { + for (let i = 0; i<=max; i++) + try { return await fetch() }
看到的是：哪几行被加了（+）、删了（-）、在哪个文件、有限的上下文（hunk header 会提示函数名）
不擅长的：稳定、结构化地理解 symbol 关系、调用链、语义影响
语言无关 零依赖 半成品代码也行 隐私友好
IDE 的方式 · LSP + AST
ClassDeclaration: ApiClient MethodDefinition: request ├─ params: [path, opts] └─ body: ForStatement └─ TryStatement └─ AwaitExpression: fetch()
看到的是：代码的结构（函数、类、参数、调用关系）
看不到的是：这次 commit 到底改了什么（它不做 diff）
需按语言安装 需语言服务 重量级
AhaDiff 的策略：用 git diff 作为真相源（"到底改了什么"），由 LLM 补全语义理解（"为什么这样改"）。未来可考虑引入 Tree-sitter AST 作为增强层（提升 symbol 提取精度），但永远不替代 diff。（AST 增强尚未纳入正式方案）

一句话：IDE 用 AST 理解代码结构，AhaDiff 用 diff 锁定变更事实 + LLM 理解变更含义。两者互补，不冲突。

Q6: 需要多个模型吗？只有一个 API key 怎么办？

BYOK 兼容
理想模式 · 双模型
生成模型（大模型如 Sonnet）：写 lesson、抽取 claim
评估模型（小模型如 Haiku）：打分、验证
优点：避免"自己给自己打高分"
退路模式 · 单模型（Fallback）
同一个模型同时做生成和评估
generate temp=0.7（发散），judge temp=0.0（严格）
设计已预留此能力，推荐仍是跨模型评估
# 单模型配置（完全可用） [llm.generate] provider = "anthropic" model = "claude-sonnet-4-5-20250514" temperature = 0.7 [llm.judge] provider = "anthropic" # 同一个 provider model = "claude-sonnet-4-5-20250514" # 同一个 model temperature = 0.0 # 但 temp=0 更严格 # 甚至可以全离线 [llm.generate] provider = "ollama" model = "qwen3-coder"
最低成本方案：用免费的本地 Ollama 模型（如 qwen3-coder）做 judge，云端大模型做 generate。这样只需一个 API key，且有跨模型评估的好处。

Q7: 所以 AhaDiff 到底是什么？一句话版本

最终定位
AhaDiff 是一个只读的本地 CLI。

它读取 AI 写出的 git diff（或 patch 文件），
把每句解释都绑定到代码证据，
标出 AI 说错的地方，
生成 Lesson / Quiz / 复习卡，
然后自动评分，越来越好。

它不改你的代码。
它教你理解你的代码。
