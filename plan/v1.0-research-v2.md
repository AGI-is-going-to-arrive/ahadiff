**🔴 第一轮误报纠正：`total_reviews` 不是 Bug**

第一轮报告声称 `scheduler_presets.total_reviews` 从未递增。第二轮深度审计发现：
- `database.py:984-991` 在 `_record_card_review()` 中正确执行 `UPDATE scheduler_presets SET total_reviews = total_reviews + 1`
- `_record_card_review` 被 `record_card_review()` 和 `record_card_review_once()` 两条路径调用
- 后者通过 `learning_signals` 表的 UNIQUE 约束实现幂等，重复 key 返回 None 不触发递增
- **结论：每次唯一 review 正确递增一次，FSRS Optimizer 不受阻塞**

### 4.4 关键约束

- **HC-G1**：不添加 `graphify` 为 Python **默认**依赖（太重：21 grammar + networkx + C 编译器）。AhaDiff 只消费 `graph.json` 静态文件
- **HC-G2**：解析器必须处理 `links`（NetworkX 格式）而非 `edges`
- **HC-G3**：节点字段是平铺的（`file_type`, `source_file`），非嵌套 `meta.type`/`meta.path`
- **SC-G1**：暴露 `confidence` tag 和 `community` ID 到 Viewer 用于信任过滤 + 聚类
- **HC-G4**：Graphify 是 v1.0 Full Scope 的一等可选能力：`full/learning_only/empty` 三态必须完整可用；未安装 Graphify 时主链路不得失败
- **HC-G5**：不提供 `ahadiff[graphify]` 依赖组，不自动安装 Graphify；只提供检测、导入、刷新、文档和 doctor 指引，避免供应链耦合

**复现性约束**：R5.4 bundle 数据可由 `pnpm run build` 复现；R5.1/R5.2/R5.3/R5.5 当前是一次性 ad-hoc baseline。v1.0 gate 前必须补固定脚本/命令，或在报告中继续标注为 ad-hoc baseline，不得作为可回归 benchmark 宣称。
